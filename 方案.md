一、核心问题诊断：全局变量网状依赖
当前框架有 14 个包级全局变量，分布在 10 个不同的包里，形成了这样的依赖网：

apps/boot/auth/init()
  → pkgauth.globalAccountFactory = func() { ... }
  → pkgauth.globalAccountAuthFactory = func() { ... }

apps/boot/tenant/init()
  → pkgtenant.globalFactory = func() { ... }

apps/rbac/user/init()
  → pkgrbac.globalUserFactory = func() { ... }

apps/rbac/role/init()
  → pkgrbac.globalRoleFactory = func() { ... }

apps/reference/weixin/module.go  (Register 闭包里)
  → pkgauth.Get()()           // nil if auth not loaded
  → pkgauth.GetAccountAuthRepository()()  // nil
  → pkgtenant.Get()()         // nil
  → pkgrbac.GetRoleRepository()()
  → pkgrbac.GetUserRepository()()
问题根源：模块的 Init() 和 Register() 分两步完成，中间没有共享的"应用上下文"。Init() 阶段各模块只能把东西推到全局变量里，Register() 阶段再从全局变量里取——这是"用全局变量模拟 DI 容器"。

方案：引入 AppContext，消除 14 个全局变量中的 10 个。

改 plugin.Module 接口，让 Init() 能接收和贡献依赖：

// framework/pkg/plugin/plugin.go

// AppContext 是模块之间共享依赖的唯一载体
type AppContext struct {
    DB         *pgxpool.Pool
    Config     *config.Config
    Session    session.SessionManager
    Authz      authz.Authorization
    Cache      *redis.Client  // 可为 nil
    
    // 模块注入的 Repository（由各模块在 Init 中填充）
    AccountRepo     auth.AccountRepository
    AccountAuthRepo auth.AccountAuthRepository
    TenantRepo      tenant.TenantRepository
    UserRepo        rbac.UserRepository
    RoleRepo        rbac.RoleRepository
    OrgRepo         rbac.OrganizationRepository
    PermRepo        rbac.RoleResourceRepository
}

type Module interface {
    Name() string
    Init(ctx *AppContext) error                        // 改动：加参数
    Register(ctx *AppContext, public, protected *gin.RouterGroup)  // 改动
    Shutdown() error
}
boot/boot.go 的 Init 变成：

func Init(cfg *config.Config) (*App, error) {
    // ... logger, db, cache 初始化不变 ...
    
    appCtx := &plugin.AppContext{
        DB:      db.Get(),
        Config:  cfg,
        Session: sm,
        Cache:   cache.Get(),
    }
    
    // 初始化每个模块时把 appCtx 传进去，模块把自己的 Repository 注入回来
    for _, m := range plugin.Apps() {
        if err := m.Init(appCtx); err != nil { ... }
    }
    
    // 现在 appCtx 上已经挂满了所有模块注入的 Repository
    // 构建 PermService 和 Authz 时可以消费它们
    permService := service.NewPermissionService(...)
    authzService := service.NewAuthorizationService(permService)
    appCtx.Authz = authz.Wrap(authzService)
    
    return &App{Context: appCtx, Server: ...}, nil
}
apps/boot/auth/module.go 变成：

func (m *module) Init(ctx *plugin.AppContext) error {
    ctx.AccountRepo = NewAccountRepository(ctx.DB)
    ctx.AccountAuthRepo = NewAccountAuthRepository(ctx.DB)
    return nil
}

func (m *module) Register(ctx *plugin.AppContext, public, protected *gin.RouterGroup) {
    deps := Dependencies{
        Config:       ctx.Config,
        AccountRepo:  ctx.AccountRepo,
        TenantRepo:   ctx.TenantRepo,
        PlatformRepo: permission.NewPlatformRoleRepository(ctx.DB),
        Session:      ctx.Session,
    }
    h := NewHandler(NewService(deps))
    Register(public, protected, h)
}
apps/reference/weixin/module.go 变成：

func (m *module) Register(ctx *plugin.AppContext, public, protected *gin.RouterGroup) {
    // 之前是 pkgauth.Get()() / pkgtenant.Get()()，现在直接从 ctx 取

    }

                      ctx.TenantRepo, ctx.RoleRepo, ctx.UserRepo)

}
收益：

删掉 framework/pkg/auth/registry.go、framework/pkg/tenant/registry.go、framework/pkg/rbac/{user,role,organization,permission}.go 中的全部全局变量
删掉对应 Register()/Get() 函数和 nil 检查
模块依赖从"隐式全局变量"变成 AppContext 上的显式字段，IDE 可追踪
init() 里不再需要调用 pkgauth.Register(...)，依赖注入统一在 Init() 完成
二、清理 ext_impl——删除停止间隙代码
framework/internal/core/ext_impl/registry.go 里有 189 行重复的类型定义和适配器——userRecord、tenantRecord、userRepoAdapter、tenantPkgAdapter。这些存在的唯一原因是 User 模块在 Phase 2 时还没完全迁出 framework/internal。

现在 User 已经在 apps/rbac/user/ 了（Phase 3），这些适配器可以全部删除。

方案：在 AppContext 上有了 UserRepo 和 TenantRepo 之后，ext_impl/provider.go 的 userFacadeImpl 和 tenantFacadeImpl 直接调用 appCtx.UserRepo 而不是通过本地适配器。

影响文件：

framework/internal/core/ext_impl/registry.go —— 整个文件删除（189 行）
framework/internal/core/ext_impl/provider.go —— 简化（从 98 行减到约 40 行）
framework/internal/core/boot/boot.go —— InitExtApi() 调用改为接收 appCtx
三、统一模块写法
当前有三种写法：

写法	模块
plugin.NewModule("name", func(...))	auth, tenant, user, role, menu, resource, permission, org, dict, asset, weixin
自定义 struct module{name} 实现接口	cms, flag
有 Init() 用 plugin.WithInit	auth, weixin
方案：全部统一为第二种自定义 struct（它更可控，方便将来加方法），但提供一个 BaseModule 减少样板：

// framework/pkg/plugin/plugin.go 新增

type BaseModule struct {
    NameStr string
    InitFn  func(ctx *AppContext) error
    RegFn   func(ctx *AppContext, public, protected *gin.RouterGroup)
    StopFn  func() error
}

func (m *BaseModule) Name() string    { return m.NameStr }
func (m *BaseModule) Init(ctx *AppContext) error {
    if m.InitFn != nil { return m.InitFn(ctx) }
    return nil
}
func (m *BaseModule) Register(ctx *AppContext, public, protected *gin.RouterGroup) {
    m.RegFn(ctx, public, protected)
}
func (m *BaseModule) Shutdown() error {
    if m.StopFn != nil { return m.StopFn() }
    return nil
}
然后 cms/module.go 从：

type module struct { name string }
func (m *module) Name() string    { return m.name }
func (m *module) Init() error     { return nil }
func (m *module) Shutdown() error { return nil }
func (m *module) Register(public, protected *gin.RouterGroup) { ... }
简化为：

func Module() plugin.Module {
    return &plugin.BaseModule{
        NameStr: "cms",
        RegFn: func(ctx *plugin.AppContext, public, protected *gin.RouterGroup) {
            h := NewHandler(ctx.DB)
            Register(h, public, protected)
        },
    }
}
同时删掉不再需要的 plugin.NewModule / plugin.NewModuleWithOpts / ModuleOption 等工厂函数，保持 API 最小化。

四、补 Service 层
cms 和 flag 目前没有 Service 层——handler 直接调 repository 或全局函数。

对 cms：加 cms/service.go，从 handler.go 中抽出业务逻辑。 对 flag：加 flag/service.go，把 handler.go（15411 行超大的 handler）中的逻辑下沉。

这不是架构问题，是代码组织问题——handler 文件越大，越容易把校验、转换、业务规则混在一起。

五、权限中间件从 internal 提升到 pkg
framework/internal/core/middleware/auth.go:209-226 里的 Require/RequireAny/RequireAll 是对 pkg/middleware 的 thin wrapper。而 pkg/middleware/auth.go 已经有一份完整实现。internal/core/middleware/auth.go 中的 wrapper 完全可以删掉，让调用方直接 import framework/pkg/middleware。

现状：

internal/core/middleware/auth.go —— Require(), RequireAny(), RequireAll(), RequireAuthenticated(), RequirePlatformRole()
pkg/middleware/auth.go —— 完全相同的五个函数
方案：删掉 internal/core/middleware/auth.go 中的这些 wrapper（保留 Auth、AuthLite、OptionalAuth、processAuthToken、injectAuthContext 这些真正的中间件逻辑），路由注册直接用 pkg/middleware.Require(...)。

八、不改的部分（明确划线）
事项	理由
保持 init() 自注册	Go 标准库也用这模式（database/sql drivers），零运行时成本
保持 PostgreSQL RLS	是核心差异化能力，换数据库等于重写
不引入 wire/fx	当前规模下，AppContext 手工注入比代码生成更直观
保持 Handler → Service → Repository 三层	符合 Go 社区主流，新人可预期
保持 internal/ 边界	Go 的 internal/ 规则是正确的架构约束
